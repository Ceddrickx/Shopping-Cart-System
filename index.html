1 — dict_Codes (Lines 8–13)

A module-level dictionary defined outside the class. It maps each valid promo code string to its integer discount percentage. Defined at module level so that any method in NUmart can access it by name without needing self.

system.py — Lines 8–13 Module-level constant — never changes at runtime
system.pyclick any line to expand
Dictionary syntax { key : value }Curly braces {} wrap the whole structure. A colon : separates each key from its value. A comma separates each pair. The extra spaces before the colons are purely for visual alignment — Python ignores them.
Why is it defined outside the class?Because it is a constant — it never changes while the program runs. Placing it at module level means every method in NUmart can access it directly by name, without needing self.. It acts as a single source of truth for all four valid codes.
Key and value typesKeys are strings (e.g., "SAVE10"). Values are integers (e.g., 10) representing the discount percentage. When Python evaluates code in dict_Codes, it checks whether that exact string is among the keys. Lookup is O(1) — instant, regardless of how many entries exist.
Accessing a value by keydict_Codes["SAVE10"] returns 10. If the key does not exist, Python raises a KeyError. This is exactly why apply_promo() checks if code in dict_Codes first — it confirms the key exists before accessing it.
2 — apply_promo() (Lines 206–222)

Validates and applies a promo code. It normalizes the user input, checks two guard conditions in sequence, sets both attributes if the code is valid, and returns True on success or False on failure. The return value is used by the retry loop in main.py to decide whether to ask for another code.

Two-layer guard systemGuard 1 (line 210): if self.promo_used: — blocks a second promo while one is already active. Guard 2 (line 215): if code in dict_Codes: — blocks invalid codes. Both conditions must pass for the promo to be applied.
Return value controls the menu loopreturn True tells main.py the code was accepted — stop asking. return False tells it the attempt failed — ask again. The while True loop in main.py checks this result with if store.apply_promo(code): break.
.strip().upper() — why both are needed.strip() removes accidental leading and trailing spaces. .upper() converts all letters to uppercase. Chained together: " save10 " becomes "SAVE10". Without this step, a valid code typed in lowercase would always be rejected.
Two attributes set at the same timeself.discount stores the percentage integer (e.g., 15), which is used in the price calculation. self.promo_used stores the code name (e.g., "STUDENT15"), which is used in display messages and as the guard flag for blocking a second code.
3 — remove_promo() (Lines 224–231)

Removes the currently active promo code. It uses a guard clause to exit early when no code is active, then resets both self.discount and self.promo_used to their default values. Both must be reset together — resetting only one would leave the system in a broken state.

system.py — Lines 224–231 Guard → confirm → reset self.discount → reset self.promo_used
system.pyclick any line to expand
Why both attributes must be reset together If only self.discount is reset to 0 but self.promo_used still holds "STUDENT15", the guard on line 210 of apply_promo() would see a non-empty string and treat it as truthy — permanently blocking any new promo code from being applied, even after the user explicitly removes it. Resetting both attributes at the same time is the only way to fully restore the default state.
4 — get_final_total() (Lines 233–238)

Computes and returns three values: the cart subtotal, the peso amount saved by the discount, and the final amount the customer must pay. It is called by display_price_summary(), process_payment(), and _generate_receipt() — so returning all three values at once avoids recomputing the same math in multiple places.

system.py — Lines 233–238 get total → compute discount amount → subtract → return all three
system.pyclick any line to expand
# Live example — STUDENT15 applied, cart total = ₱200.00
compute_total()total = 200.00# loops cart: price × qty per item
15 / 100= 0.15# convert % to a decimal multiplier
200.00 × 0.15discounted = 30.00# amount saved (the deduction)
200.00 − 30.00final = 170.00# ₱170.00 — what the customer pays
Why return three values?display_price_summary() needs all three to show the full breakdown. process_payment() needs only final for the Amount Due display. _generate_receipt() needs all three for the receipt printout. Returning them together avoids repeating the same calculation.
What if no promo is active?When self.discount is 0: discounted = total × (0 / 100) = 0 and final = total − 0 = total. The customer pays full price. No special case is needed — the math works correctly either way.
5 — display_price_summary() (Lines 240–262)

Formats and prints the full price breakdown to the terminal. It first guards against an empty cart, then calls get_final_total() to retrieve all three computed values, formats them as peso strings with two decimal places, and conditionally prints the promo and discount rows only when a discount is active.

system.py — Lines 240–262 Guard → get totals → format strings → print (with optional promo rows)
system.pyclick any line to expand
Guard clause firstLines 242–244 check whether the cart is empty before doing anything else. If it is, a warning is printed and return exits the function immediately. This prevents a confusing ₱0.00 price summary from appearing when the cart has nothing in it.
f-string :.2f format specifierf"₱{total:.2f}" — the :.2f part means "format this number as a fixed-point decimal with exactly two decimal places." Without it, Python might print 170.0 or even 170.00000001 due to floating-point imprecision.
discounted_str is always computeddiscounted_str is formatted on line 248 regardless of whether a promo is active — but it is only printed inside the if self.discount > 0 block on line 256. When no promo is active, the string is computed but never displayed.
Calls get_final_total() on line 246The result is unpacked into three separate variables in one statement: total, discounted, final = self.get_final_total(). This is Python's multiple assignment — the function returns a tuple of three, and they are split into named variables instantly.
