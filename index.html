1 — dict_Codes (Lines 8–13)
system.py — Lines 8–13 Module-level constant — never changes at runtime
system.py↓ click any line to explain
Dictionary syntax { key : value }Curly braces wrap the whole thing. Colon separates key from value. Comma separates pairs. Extra spaces before : are just alignment — Python ignores them.
Why outside the class?It's a module-level constant that never changes. All methods in NUmart access it by name directly — no self needed. One source of truth for all 4 codes.
Key and value typesKeys are str ("SAVE10"). Values are int (the discount %). When you do code in dict_Codes, Python checks if that exact string is a key. O(1) — instant, no loop needed.
Accessing valuesdict_Codes["SAVE10"] → 10. The bracket notation retrieves the value for that key. If the key doesn't exist, you'd get a KeyError — which is why we check if code in dict_Codes first.

.strip().upper() — both are needed.strip() removes accidental spaces from both ends. .upper() converts all letters to uppercase. Chained together: " save10 " → "SAVE10". Without this, valid codes in lowercase would always fail.
Two attributes set togetherself.discount stores the number (e.g. 15). self.promo_used stores the code name (e.g. "STUDENT15"). Discount is used in math. promo_used is used in display messages and as the guard flag.

Why return 3 values?display_price_summary() needs all 3 for the breakdown. process_payment() (line 270) needs final for Amount Due. _generate_receipt() (line 404) needs all 3 for the receipt. Returning once avoids recomputing the same math three times.
What if discount = 0?discounted = total × (0/100) = 0. final = total − 0 = total. No promo = no discount. The math always works correctly — no special case needed.
